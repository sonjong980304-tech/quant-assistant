# DART Text-to-SQL + LLM Wiki + 백테스트 — 아키텍처 정리 (면접 대비)

> 한 줄 요약: **"한국 상장사 재무·주가 데이터를 모아 SQLite에 넣고, 자연어 질문을 LLM이 SQL로 바꿔 답하며(LangGraph), 그 SQL을 3층으로 평가하고, 같은 데이터로 퀀트 백테스트까지 돌리는 풀스택 시스템"**

---

## 1. 전체 그림 (먼저 이걸 머릿속에 그려라)

```
[데이터 수집]                [백엔드 = 두뇌]                  [프론트엔드]
DART OpenAPI ─┐                                            
pykrx 주가   ─┼─→ SQLite(market.db) ─→ FastAPI ─→ 단일 HTML(바닐라 JS)
KSIC 업종    ─┘        ↑                  │  ├ /api/query  → LangGraph 파이프라인
                       │                  │  ├ /api/eval   → 3층 평가
                       └──────────────────┘  └ /api/backtest → 백테스트 엔진
```

세 덩어리로 나눠서 말하면 된다:
1. **데이터 계층** — 외부 API에서 긁어와 SQLite에 정규화해 저장
2. **백엔드(두뇌)** — ① 자연어→SQL(LangGraph) ② SQL 평가 ③ 백테스트
3. **프론트엔드** — FastAPI가 주는 JSON을 받아 그리는 단일 페이지

기술 스택: **Python · FastAPI · LangGraph · SQLite · OpenAI/Ollama · 바닐라 JS(프레임워크 없음)**

## 2. 데이터 계층 (`src/ingest/`)

외부 데이터를 긁어 **SQLite 한 파일(`data/market.db`)** 에 넣는다. ORM 없이 순수 SQL.

| 소스 | 가져오는 것 | 파일 |
|---|---|---|
| **DART OpenAPI** | 재무제표(매출·영업이익·순이익·자본 등), 상장주식수 | `dart.py` |
| **pykrx** | 일별/월말 주가 | `krx.py`, `price_live.py` |
| **KSIC 표준산업분류** | 업종(섹터)·시장 매핑 | `ksic.py` |

핵심 테이블:
- `company` (종목코드, 이름, 시장, 섹터)
- `financials` (종목코드, 분기, 계정키, 금액) ← **세로(long) 구조**: 한 행 = 한 계정
- `prices` (종목코드, 날짜, 종가, 시가총액)
- `delisting` (상장폐지 정보 — 백테스트 생존편향 방지용)
- `wiki` (질의 **기록 로그**), `backtest_runs` (백테스트 결과)

> 면접 포인트: financials를 `(account_key, amount)` **세로 구조**로 둔 이유 = 회사·분기마다 제공되는 계정이 달라서, 컬럼 고정(가로) 대신 키-값으로 유연하게. 대신 조회 시 `MAX(CASE WHEN account_key='revenue' THEN amount END)` 식 **피벗**이 필요.

## 3. 백엔드 핵심 ① — 자연어 → SQL (LangGraph)

### 3-1. "딥에이전트"가 아니라 LangGraph StateGraph다 (중요)

면접에서 헷갈리지 마라:
- **DeepAgents(딥에이전트)** = LangChain의, 스스로 계획 세우고 도구를 자유롭게 반복 호출하는 자율 에이전트. **이 프로젝트는 안 썼다.**
- **이 프로젝트** = **LangGraph의 `StateGraph`** = 노드(함수)들을 **정해진 순서의 그래프**로 연결한 파이프라인. 에이전트가 멋대로 판단하는 게 아니라 **흐름이 고정**돼 있어 예측 가능하고 디버깅이 쉽다.

> 왜 자율 에이전트가 아니라 고정 그래프? → Text-to-SQL은 단계(정제→라우팅→생성→실행→평가)가 명확해서, 자율성보다 **안정성·재현성**이 중요하기 때문.

### 3-2. 그래프 흐름 (`src/graph/build.py`)

```
refine → router → sql_gen → execute → ⟦diagnose⟧ →┬ none(정상)  → record → (평가) → END
            ↑___________________________________│  ├ sql 문제    → sql_gen 재시도(피드백)
            └───────────────────────────────────┤  ├ refine 문제 → 질문 재정제
                                                 └ data/3회초과 → human_review(HITL) → END
```

`StateGraph(GraphState)`에 노드를 등록하고 엣지로 연결. `add_conditional_edges`로 **진단 결과(원인)에 따라 분기**(통과/재시도/사람검토)하고, 평가 여부로도 분기한다. → **self-correcting + human-in-the-loop** 그래프.

### 3-3. 공유 상태 = GraphState (`src/graph/state.py`)

LangGraph의 핵심 개념. **모든 노드가 하나의 딕셔너리(state)를 주고받는다.** 각 노드는 state를 읽고, 바뀐 부분만 반환하면 LangGraph가 자동 병합한다.

```python
class GraphState(TypedDict, total=False):
    raw_question / question      # 원본 질문 / 정제된 질문
    route / data_version         # 라우팅 결과 / 데이터 버전
    sql / sql_source             # 생성된 SQL / 출처('generated'|'fallback')
    rows / columns / row_count / error   # 실행 결과
    diagnosis / needs_human / attempt_count  # 진단 결과 / HITL 필요 / 재시도 횟수
    evaluation / notes           # 평가 결과 / 단계별 로그
```

### 3-4. 노드 8개 (`src/graph/nodes.py`) — 하나씩 설명

| # | 노드 | 하는 일 | 쉽게 |
|---|---|---|---|
| 1 | **refine_node** | LLM이 질문을 깔끔히 다듬음 ("삼전 per" → "삼성전자 PER") | 오타·줄임말 정리 |
| 2 | **router_node** | 재무/주가/둘다 판단(`financial`/`price`/`both`) + '오늘 주가'류면 실시간 보강 | 어느 테이블 볼지 |
| 3 | **sql_gen_node** | 정제 질문 + 스키마 + 예시 → LLM이 SQL 생성 | 자연어→SQL |
| 4 | **execute_node** | 생성된 SQL을 SQLite에서 실행 | 답 가져옴 |
| 5 | **diagnose_node** | 결과가 이상하면(에러·빈결과·개수부족·이상치) 원인 진단 | **결과가 말이 되나 검사** |
| 6 | **record_node** | 성공한 질의를 기록(로그) | 저장 |
| 7 | **human_review_node** | 자동으로 못 고치면(데이터문제/3회초과) 사람 검토(HITL) | 사람 호출 |
| 8 | **eval_node** | (평가 모드) 3층 평가 | 채점 |

#### 자가 수정은 2층 구조다 (면접 하이라이트 ⭐)

**1층 — 실행 에러 자가수정 (sql_gen 내부)**: SQL을 만들고 즉시 실행 → SQLite 에러(문법·없는 컬럼) 나면 에러 메시지를 LLM에 되먹여 **최대 3회 재시도**, 실패 시 휴리스틱 폴백. → **"SQL이 죽었나"** 를 잡음.

**2층 — 진단 서브에이전트 (diagnose_node)**: 1층은 "SQL이 실행됐나"만 본다. 하지만 **SQL이 멀쩡히 실행됐는데 결과가 말이 안 되는** 경우(빈결과, 5개 요청에 2개, ROE 1398% 같은 이상치)는 못 잡는다. diagnose_node가 이걸 잡는다.

**핵심 원리: 증거는 코드, 해석은 LLM(gpt-5.5).**
LLM에게 그냥 "왜 결과가 이상해?"라고 물으면 데이터를 못 보니 환각한다. 그래서:
1. **코드가 증거 수집** — 결과 상태 분류(에러/빈결과/개수부족/이상치) + **조건별 분해**(WHERE를 하나씩 빼며 재실행해 어느 조건에서 행이 급감하는지) + sanity 체크(ROE>200%·시총>3000조)
2. **gpt-5.5가 분류** — 증거를 받아 원인을 4가지로: `sql`(SQL 오류→자동수정) / `refine`(질문 모호→재정제) / `data`(데이터 미수집·오류→사람) / `none`(사실 정상)
3. **라우팅** — sql/refine이면 피드백 주고 재시도(최대 3회), data거나 3회 초과면 **human_review(HITL)** 로 전 과정+진단을 사람에게

**실제 예 (이거 말하면 강함):**
"반도체 PER 5개"가 2개만 나왔을 때 → 코드가 조건별 분해 → `market_cap IS NOT NULL`에서 5→2 급감 포착 → gpt-5.5가 "시총 데이터 미수집(data 문제), SQL 고쳐도 복구 안 됨"으로 정확 분류 → 사람에게. **같은 '적은 결과'라도 "조건이 빡세서 N개가 답(none)"인지 "데이터 누락(data)"인지를 증거로 구분**한다. (왜 LLM 혼자 안 되냐 → 데이터를 직접 못 보니 환각하기 때문. 그래서 코드가 증거를 만들어 준다.)

### 3-5. LLM 추상화 (`src/llm.py`)

같은 인터페이스로 **두 종류 모델**을 바꿔 끼움:
- **OpenAI (GPT-5.4-mini 등)** — 클라우드, 성능 좋음
- **Ollama (Qwen2.5-coder:7b, 4bit 양자화)** — 로컬 무료, 프라이버시

> 면접 포인트: provider를 설정(`.env`)으로 갈아끼우는 **전략 패턴**. SQL 생성/판단(judge)별로 다른 모델 역할(role) 지정 가능.

### 3-6. 프롬프트 엔지니어링 (`src/graph/prompts.py`)

여기가 정확도의 80%. 들어가는 것:
- **스키마 설명** (테이블·컬럼·구조)
- **지표 계산 규칙** (PER=시총/순이익TTM, ROE=순이익TTM/자기자본 등)
- **few-shot 예시** (질문→정답 SQL 쌍 몇 개)
- **가드 규칙** (예: ROE는 `순이익<자기자본` 조건으로 자본잠식 이상치 제외, 랭킹은 거래 중 상장사만 등 — 실데이터 이상치를 막는 노하우)

## 4. 백엔드 핵심 ② — 평가 (`src/eval/`)

생성된 SQL이 **진짜 맞는지 3층(Layer)으로 채점**. (`evaluator.py`)

| 층 | 이름 | 무엇을 보나 | 어떻게 |
|---|---|---|---|
| **Layer 3** | 실행 가능성 | 문법이 안전한 SELECT인가 / 실행되나 / 결과가 비었나 | 규칙 체크 (LLM 불필요) |
| **Layer 1** | Execution Accuracy (EX%) | **생성 SQL 결과 == 정답 SQL 결과인가** | denotation 비교 |
| **Layer 2** | LLM-as-Judge | SQL이 질문에 적절한가 (1~5점) | 다른 LLM이 채점 |

### 핵심 개념: denotation accuracy (Layer 1)
"SQL 텍스트가 똑같은가"가 아니라 **"실행 결과(값)가 같은가"** 로 비교한다. 그래서:
- **컬럼 이름·별칭·순서는 무시**하고 **값만** 비교 (`compare_resultsets`)
- 상위 N 정렬 질문은 **행 순서까지** 비교(order_sensitive), 아니면 multiset(순서 무관)

> 왜? 정답 SQL과 생성 SQL이 문법은 달라도(`SELECT a,b` vs `SELECT b,a`) 답이 같으면 정답으로 쳐야 공정하니까. Text-to-SQL 논문(Spider 등)의 표준 평가 방식.

### 정답셋 (`src/eval/goldset.py`)
손으로 만든 **질문 + 정답 SQL 50쌍**. `runner.py`가 전체를 돌려 **EX%(정확도), Judge 평균점수, Layer3 통과율**을 리포트로 뽑는다.

## 5. 백엔드 핵심 ③ — 백테스트 (모듈 B, `src/backtest/`)

같은 데이터로 **퀀트 전략 시뮬레이션**. (예: "저PER+고ROE 10종목, 분기 리밸런싱")

| 파일 | 역할 |
|---|---|
| `data_access.py` | **시점별 지표 계산**(`metrics_at`) — 과거 각 시점에 PER/ROE 등을 그때 데이터로 |
| `selection.py` | 종목 선정 — z-score / 순위합 / 교집합 방식 |
| `engine.py` | 리밸런싱 루프 — 사고팔고, 거래비용 반영, NAV(수익곡선) 계산 |
| `performance.py` | 성과지표 — 누적수익·CAGR·MDD·샤프·**벤치마크 대비 초과수익·베타** |

### 면접에서 빛나는 3가지 "퀀트 함정 방지"
1. **Look-ahead 방지(미래 참조 금지)**: 2024년 시점 백테스트에 2024년 5월 공시된 재무를 쓰면 반칙. → `disclosed_date <= 시점` 인 데이터만 사용.
2. **생존편향 제거(survivorship bias)**: 지금 살아남은 회사만 보면 수익이 부풀려짐. → `delisting` 테이블로 그 시점에 **상장폐지 전이던 종목만** 포함.
3. **거래비용 반영**: 수수료·세금·슬리피지를 회전율만큼 차감.

> 실제 디버깅 사례(면접에서 말하면 좋음): 백테스트가 스팩·데이터오류·상폐 종목을 뽑아 "−1%"라는 가짜 결과를 냈는데, 이상치 가드(거래종목만·스팩제외·ROE<100% 등)를 넣자 진짜 결과(−13%, 시장 대비 −47%)가 드러났다. → **"좋아 보이는 숫자가 사실은 데이터 쓰레기였다"** 는 이야기.

## 6. 프론트엔드 (`web/static/index.html`)

- **단일 HTML 파일 + 바닐라 JavaScript**. React/Vue 같은 프레임워크 **안 씀**.

> 면접 포인트: 왜 프레임워크 없이? → 화면이 단순하고 의존성을 줄이려고. "가벼운 단일 페이지면 바닐라로 충분하다"는 판단. (규모 커지면 React 등으로 갈 수 있음을 덧붙이면 균형 잡힌 답)

## 8. 면접 예상 질문 & 모범 답변

**Q. 이 프로젝트 한 줄로?**
> 한국 상장사 재무·주가를 모아, 자연어 질문을 LLM이 SQL로 바꿔 답하고(LangGraph), 그 SQL을 3층으로 평가하며, 같은 데이터로 퀀트 백테스트까지 하는 풀스택 데이터 시스템입니다.

**Q. LangGraph를 왜 썼나? 그냥 LLM 한 번 부르면 안 되나?**
> 자연어→SQL은 정제→라우팅→생성→실행→평가로 단계가 나뉘고, 각 단계의 상태를 공유·추적해야 합니다. LangGraph의 StateGraph로 노드를 명시적 그래프로 묶으면 흐름이 고정돼 **재현성과 디버깅**이 좋고, 실행 실패 시 에러를 다시 LLM에 넣는 **self-correction 재시도**도 노드 안에 자연스럽게 넣을 수 있습니다.

**Q. 딥에이전트(자율 에이전트)는 왜 안 썼나?**
> Text-to-SQL은 절차가 명확해서 자율성보다 **안정성**이 중요합니다. 에이전트가 도구를 자유롭게 반복 호출하면 비결정적이고 비용·지연이 커집니다. 그래서 고정된 그래프 + 폴백(휴리스틱)으로 예측 가능하게 설계했습니다.

**Q. LLM이 틀린 SQL을 만들면?**
> ① 만든 SQL을 즉시 실행 검증 → ② 에러를 LLM에 피드백해 최대 3회 재시도 → ③ 그래도 실패하면 규칙 기반 휴리스틱 SQL로 폴백합니다. 시스템이 죽지 않습니다.

**Q. SQL 정확도를 어떻게 측정하나?**
> 3층입니다. Layer3는 실행 가능성(안전 SELECT·실행·빈결과), Layer1은 정답 SQL과 **결과값 비교(denotation, 컬럼명·순서 무시)**로 EX%를 내고, Layer2는 다른 LLM이 적절성을 1~5점으로 채점합니다. 정답셋 50문항으로 배치 평가합니다.

**Q. 가장 어려웠던 점 / 디버깅 경험은?**
> 실데이터의 이상치였습니다. DART 원본의 단위 오류(주식수 ×100만), 적자/자본잠식 회복 종목의 ROE 폭발(1,398%), 상장폐지·스팩 종목 혼입 등이 랭킹과 백테스트를 오염시켰습니다. SQL 프롬프트와 백테스트 엔진 양쪽에 **이상치 가드**(거래 중 상장사만·ROE<100%·순이익 정합성 등)를 넣어 해결했고, 특히 백테스트의 "좋아 보이던 −1%"가 사실은 데이터 쓰레기였고 진짜는 시장 대비 −47%였다는 걸 밝혀낸 게 인상적이었습니다.

**Q. 데이터 신뢰성은 어떻게 지키나?**
> look-ahead 방지(공시일 기준), 생존편향 제거(상폐 테이블), 거래비용 반영 등 퀀트 표준 함정을 코드로 막고, 단위·정합성 가드로 원본 오류를 걸러냅니다.

## 9. 키워드 암기 카드

- **LangGraph / StateGraph**: 노드(함수)를 그래프로 연결한 파이프라인. 상태(dict) 공유.
- **노드 6개**: refine(정제)→router(라우팅)→sql_gen(생성)→execute(실행)→record(기록)→eval(평가)
- **self-correction**: 에러를 LLM에 되먹여 3회 재시도, 실패 시 휴리스틱 폴백
- **denotation accuracy (EX%)**: SQL 텍스트가 아니라 **결과값**으로 정답 판정
- **3층 평가**: L3 실행가능성 / L1 결과정확도 / L2 LLM판정(1~5)
- **백테스트 3대 방지**: look-ahead / 생존편향 / 거래비용
- **이상치 가드**: 단위오류·ROE폭발·상폐·스팩 제거
- **스택**: FastAPI · LangGraph · SQLite · OpenAI/Ollama · 바닐라 JS
